# ADC采集信号测试

将当前主程序切换为单 ADC 连续采集和 VOFA+ 输出。开始前读取 `STATUS.md`、`README.md`、`GUIDE.md`，并执行 `git status --short`。

## 输入与安全边界

- 要求用户明确指定 MCU 引脚、ADC 实例或通道、预计电压范围、采样率、串口端口和波形是否带直流偏置。
- 不猜测 ADC 通道或板级接线；从 `.ioc`、生成的 `adc.c` 和芯片 HAL 定义确认映射。
- 片内 ADC 引脚只接受 0～3.3 V。双极性信号必须先经偏置、限幅和共地调理；先用示波器确认范围后再接入。
- 检查与当前主任务、AD9833、DAC8830、双 ADC 相位任务、TIM、DMA、UART 的引脚及资源冲突。只初始化本任务必需外设。

## 实现流程

1. 在 `HDL/` 或生成外设层确认 ADC 通道、采样时间、触发源和 DMA 配置；业务采集调度放 `FML/`，应用入口放 `API/`。
2. 使用循环 DMA，采用 2 的幂帧长（默认 256 或 512 点）。缓冲区放 `.dma_buffer`、32 字节对齐；D-Cache 开启时，在 CPU 读取 DMA 写入区域前执行按范围失效操作。
3. 在 DMA 半传输或全传输回调中只置位或登记已完成半块；在主循环或受控发送路径中处理并发送，避免在中断内阻塞 UART。
4. 将原始码按 ADC 实际分辨率和 Vref 换算为伏特。FireWater 每帧只输出一个字段，例如 `adc_pa1:%.6f`；不要混入普通日志、提示文本或不同格式的数据。
5. 串口使用 DMA 或非阻塞发送。若上一帧尚未发送完成，丢弃最新待发帧并计数，不得阻塞采样或覆盖 DMA 正在写入的半块。
6. 在 `main.c` 仅启动 GPIO、DMA、所选 ADC、触发定时器（若使用）和目标 UART，调用 API 初始化与循环处理。新增 `.c` 文件加入根目录 `CMakeLists.txt`。

## 参数与吞吐量

- 明确区分 ADC 采样率、每帧点数和 UART 波特率。FireWater 文本输出速率必须低于串口有效带宽；默认以 115200 baud 输出时，优先 100～1000 点每秒级可视化，不要盲目逐点转发 1 MS/s。
- 若用户需要高采样率，保持 ADC 采样率不变，但抽取后输出；在代码和 VOFA+说明中标明抽取比与有效显示采样率。
- 在 FireWater 数据流中不插入文本；状态信息改为计数器并按需由调试器查看。

## 验证与交付

1. 编译：`cmake --build --preset Debug`，检查 FLASH、RAM 占用和 `git diff --check`。
2. 用户要求上板时才烧录。先以安全直流值验证零点和比例，再用低频正弦验证频率、峰峰值和直流偏置。
3. VOFA+ 选择 `FireWater`，串口参数与固件一致；确认出现预期字段名并连续刷新。
4. 报告已编译、已烧录、已上板和已验收的边界；记录引脚、通道、采样率、帧长、抽取比、串口配置、丢帧计数和待复测项到 `STATUS.md`。同步更新 `GUIDE.md` 的接线和 VOFA+ 使用说明。

## 最小验收标准

- 连续运行期间无 ADC DMA 错误，DMA 半块没有被 CPU 或 UART 发送路径改写。
- VOFA+ 持续接收单一 FireWater 字段，波形频率、幅值和偏置与示波器在合理误差内一致。
- 关闭或切换任务时停止 ADC DMA、定时器和 UART DMA，避免占用后续主任务资源。
